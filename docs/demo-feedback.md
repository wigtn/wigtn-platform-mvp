# FIELDNOTE 입찰용 데모 피드백 정리

작성 기준: 2026-07-24 20:11 KST  
대상 코드베이스: `/Users/wigtn/wigtn-server/services/fieldnote`

## 1. 총평

현재 FIELDNOTE는 기능 범위는 많이 들어가 있지만, **입찰 평가자가 무엇을 검증하고 있는지 UI가 충분히 설명하지 못합니다.**

특히 아래 두 가지가 가장 큰 문제입니다.

1. **역할/권한 체계가 UI만 봐서는 헷갈림**
2. **AI 답변 생성 기능이 실제 호출인지 더미/fallback인지 명확하지 않음**

입찰용 데모는 일반 서비스 UI와 달리 “무엇이 실제 구현이고 무엇이 데모 시뮬레이션인지”를 명확히 보여줘야 합니다. 현재는 기능들이 예쁘게 배치되어 있지만, 평가자가 신뢰할 수 있는 설명 레이어가 약합니다.

## 2. P0 — 역할/권한 체계가 불명확함

### 현재 상태

코드상 역할은 4개입니다.

- 비회원
- 일반 영업인
- 인증 영업인
- 운영 관리자

역할 설명도 존재합니다.

| 역할 | 현재 설명 |
|---|---|
| 비회원 | 회사 리뷰와 공개 커뮤니티를 읽습니다. |
| 일반 영업인 | 프로필·게시글·답변 작성 흐름을 체험합니다. |
| 인증 영업인 | 재직 확인 배지가 붙는 리뷰와 답변을 작성합니다. |
| 운영 관리자 | 리뷰 검수·회원 인증·콘텐츠 운영 화면을 엽니다. |

### 문제

역할 설명이 너무 요약형이라, 실제로 어떤 기능이 가능하고 불가능한지 알기 어렵습니다.

사용자는 다음을 헷갈릴 수 있습니다.

- 비회원은 리뷰 작성이 가능한가?
- 일반 영업인은 질문만 가능한가, 답변도 가능한가?
- 인증 영업인은 일반 영업인과 무엇이 다른가?
- 운영 관리자는 일반 사용자 기능도 쓸 수 있는가?
- 데모 계정 로그인과 역할 전환은 다른 것인가?

### 개선 제안

첫 방문 모달 또는 상단 `역할 변경` 패널에 권한 매트릭스를 추가합니다.

예시:

| 기능 | 비회원 | 일반 영업인 | 인증 영업인 | 운영 관리자 |
|---|---:|---:|---:|---:|
| 회사 리뷰 열람 | 가능 | 가능 | 가능 | 가능 |
| 익명 리뷰 작성 | 제한 | 가능 | 가능 + 인증 표시 | 검수 가능 |
| Q&A 질문 작성 | 불가 | 가능 | 가능 | 가능 |
| 현직자 답변 작성 | 불가 | 가능 | 인증 배지 포함 | 관리 가능 |
| 프로필 수정 | 불가 | 가능 | 가능 | 가능 |
| 배지 신청 | 불가 | 가능 | 인증 완료 상태 | 승인/반려 |
| 관리자 콘솔 | 불가 | 불가 | 불가 | 가능 |
| 회사 데이터 관리 | 불가 | 불가 | 불가 | 가능 |
| 리뷰/콘텐츠 블라인드 | 불가 | 불가 | 불가 | 가능 |

## 4. P0 — AI 실제 호출/Fallback 상태가 불명확함

### 현재 코드상 동작

`lib/demo-store.ts` 기준으로 AI는 두 모드를 가집니다.

#### Live 경로

Supabase 설정이 있으면:

1. `recordAction("ai.answer.request", question)`
2. Supabase private queue 등록
3. 맥미니 worker 처리
4. OpenAI Responses API 호출
5. `recordAction("ai.answer.poll", { requestId })`로 polling
6. 답변 표시

#### Fallback 경로

Supabase 설정이 없으면:

```ts
model: "demo-fallback"
answer: FIELDNOTE_AI_FALLBACK
```

즉, 실제 AI 호출 없이 고정 샘플 답변을 반환합니다.

### 현재 로컬 환경 확인

- 루트 `.env.local`: 없음
- `backend/.env`: 있음
- `backend/.env`에는 `OPENAI_API_KEY` 존재

따라서 현재 로컬 프론트에서는 Supabase 공개 설정이 없어 `demo-fallback`으로 동작할 가능성이 큽니다.

### 문제

UI는 Live인지 Fallback인지 명확히 보여주지 않습니다.

현재 AI 진행 UI는 다음처럼 실제 호출처럼 보입니다.

- 질문 내용 검사
- AI 초안 작성
- 커뮤니티 답변 대기
- “상황을 정리해 바로 실행할 답변을 만들고 있습니다.”
- “대개 10초 안팎이 걸립니다.”

하지만 fallback이면 900ms 후 고정 답변이 반환됩니다.

최종 답변 footer에 `{model}`이 표시되기는 하지만, `demo-fallback`의 의미가 설명되지 않고 너무 약합니다.

### 개선 제안

질문 작성 폼 상단과 AI 진행 화면에 명확한 실행 상태 패널을 추가합니다.

Live일 때:

```txt
AI 모드: 실제 호출
Provider: OpenAI Responses API
Queue: Supabase private queue
Worker: Mac mini worker
Guardrail: 입력/출력 안전성 검사 적용
Data scope: 방문자별 데모 세션
```

Fallback일 때:

```txt
AI 모드: Demo fallback
실제 AI API를 호출하지 않습니다.
이 환경에는 Supabase 공개 설정이 없어 고정 샘플 답변으로 UI 흐름만 체험합니다.
```

답변 카드에도 배지를 붙입니다.

- `LIVE AI`
- `DEMO FALLBACK`
- `MOCK RESPONSE`
- `GUARDRAIL PASSED`

## 5. P0 — 기능별 실제/더미/dry-run 경계가 안 보임

### 문제

FIELDNOTE 데모는 실제 구현과 시뮬레이션이 섞인 구조입니다.

예시:

| 기능 | 현재 성격 |
|---|---|
| 회사/리뷰/커뮤니티 | seed + overlay 또는 Supabase action ledger |
| AI 답변 | Supabase 설정 시 Live, 없으면 fallback |
| 관리자 명령 | harmless demo command 성격 |
| XLSX 검사 | 샘플 dry-run |
| 크롤러 후보 | crawler dry-run, 실제 크롤링 아님 |
| 업로드/증빙 | 합성 상태 |

하지만 UI에서는 이 경계가 잘 드러나지 않습니다.

### 개선 제안

기능 카드마다 라벨을 붙입니다.

| 라벨 | 의미 |
|---|---|
| `LIVE` | 실제 DB/API/worker 호출 |
| `DEMO DATA` | 샘플 데이터 기반 |
| `DRY RUN` | 저장/외부 작업 없이 사전검사만 시뮬레이션 |
| `FALLBACK` | 외부 설정이 없을 때 고정 응답 |
| `POST-CONTRACT` | 수주 후 adapter 교체 예정 |

예시:

- AI 답변: `LIVE` 또는 `FALLBACK`
- XLSX 검사: `DRY RUN`
- 크롤러 후보: `DRY RUN · 실제 크롤링 없음`
- 배지 신청: `DEMO DATA · 실제 증빙 업로드 없음`

## 7. P1 — 데모 체크리스트/추천 체험 순서 필요

### 문제

README에는 체험 흐름이 잘 정리되어 있지만, UI 안에는 없습니다.

입찰 평가자는 문서를 먼저 읽지 않고 화면부터 만질 가능성이 큽니다. 그러면 무엇을 눌러야 전체 범위를 검증할 수 있는지 약합니다.

### 개선 제안

홈 또는 상단 패널에 `입찰 평가용 추천 체험 순서`를 추가합니다.

예시:

```txt
입찰 평가용 추천 체험 순서
1. 비회원으로 회사 리뷰 탐색
2. 일반 영업인으로 질문 작성
3. AI 답변 생성 확인
4. 인증 영업인으로 배지 표시 확인
5. 운영 관리자로 리뷰 블라인드
6. 공개 화면에서 반영 확인
```

각 항목에 다음을 표시합니다.

- 시작 버튼
- 필요한 역할
- 실제/데모 여부
- 완료 여부

## 8. P1 — 관리자 액션의 신뢰 장치 부족

### 문제

관리자 기능은 버튼을 누르면 상태가 바뀌지만, 운영 시스템의 신뢰 장치가 약하게 보입니다.

부족한 정보:

- 누가 처리했는가
- 어떤 권한으로 처리했는가
- 처리 전/후 상태가 무엇인가
- 감사 로그가 남는가
- 되돌릴 수 있는가
- 공개 화면에 어떻게 반영되는가

### 개선 제안

관리자 액션 후 receipt를 표시합니다.

예시:

```txt
처리 완료
Action: review.blind
Actor: FIELDNOTE 운영팀
Target: 리뷰 #r3
Result: 공개 피드에서 제외됨
Audit: 데모 감사 로그에 기록됨
Undo: 복구 가능
```

## 11. P2 — XLSX/크롤러 dry-run 안내 강화

### 문제

관리자 회사 데이터 화면에서 XLSX 검사와 크롤러 후보는 실제 작업처럼 보일 수 있습니다.

하지만 현재 데모에서는:

- XLSX는 샘플 검사
- 크롤러는 실제 크롤링이 아니라 후보 미리보기 dry-run

### 개선 제안

각 기능 카드에 명확한 라벨을 붙입니다.

```txt
XLSX 검사
DRY RUN · 실제 파일 저장 없음
```

```txt
외부 회사 후보 미리보기
DRY RUN · 실제 크롤링 실행 없음 · 자동 공개 0건
```

## 13. 한 줄 결론

FIELDNOTE는 기능 커버리지는 입찰 데모로 꽤 괜찮지만, **평가자가 신뢰할 수 있게 설명하는 UI**가 부족합니다. 특히 **권한 체계와 AI 호출 상태가 불투명해서**, 실제로 사용해보는 사람은 “이게 진짜 구현인지, 예쁜 샘플인지” 헷갈릴 수 있습니다.
